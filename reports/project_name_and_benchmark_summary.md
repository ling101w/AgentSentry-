# 参赛作品名称与 Benchmark 实验统计

更新时间：2026-07-03

## 推荐参赛名称

**玄鉴**

副标题：

**面向智能体工具调用链的实时行为监督与风险拦截系统**

## 命名说明

“玄”强调复杂、深层、隐蔽的智能体风险链路，“鉴”对应鉴别、审计、监督和告警。这个名字适合作为独立参赛作品名称，也能体现系统对工具调用链进行实时观察、判定和拦截的定位。

正式展示时可以这样写：

> 玄鉴：面向智能体工具调用链的实时行为监督与风险拦截系统

如果需要英文副标题，可使用：

> XuanJian: Real-time Behavior Supervision and Risk Interception for Agent Tool-Calling Chains

## Benchmark 实验总体情况

本轮实验共完成两批公开 benchmark 适配评测，合计 **840 条次**本地 `/command-lab` 映射回归。所有样例均通过 `http://127.0.0.1:8765/api/lab/command` 进入真实策略链，但样例和标签来自公开数据并参与过开发，因此不能称为独立盲测，也不能称为 AgentDojo/InjecAgent 原生运行。

危险命令没有直接落地破坏主机，实验评估的是系统在工具执行前是否能做出正确的拦截、审批或放行决策。

## LLM-Judge 使用情况

最新推荐结果采用 DeepSeek-compatible `risk-tiered` LLM-Judge：

- benchmark 入参统一使用 `semanticJudge=default`。
- 服务端当前 `semantic.mode=risk-tiered`。
- 低风险请求主要由确定性策略、TaskSpec、污点传播、ABAC、Memory Guard 和 System Preflight 处理。
- 敏感路径、外部 sink、非平凡 shell、记忆写入、持久化、工具劫持、基础文件风险才调用 LLM-Judge。

对照实验中 full LLM-Judge 也已保留：它同样做到高风险漏放 0，但延迟更高，且出现 7 条正常样例误拦。这说明 LLM-Judge 更适合作为高风险路径强复核，而不是无差别覆盖所有请求。

## 本轮真实实验结论

本轮 risk-tiered 两批公开 benchmark 合计完成 **840 条**样例评测：

- 攻击/期望保护：711 条
- 正常/期望放行：129 条
- 高风险漏放：0 条
- 误拦：0 条
- Harness error：0 条

结论是：玄鉴对高风险工具调用、数据外泄、MCP 工具攻击、记忆投毒、RCE/命令类风险拦截稳定；risk-tiered 相比 full LLM-Judge 降低了延迟和误拦，更适合作为生产化模式。

这个结果适合在报告中如实表述为：系统安全边界强，公开样例中未出现高风险漏放；风险分层语义复核在保持安全性的同时降低误判和成本；后续生产化仍需继续优化审批缓存、真实业务白名单和 TaskSpec 边界表达。

## 生产化优化方向

后续优化不应按 benchmark 关键词补丁式加规则，而应围绕真实智能体运行时的“授权、上下文、数据流、工具副作用”建立更清晰的治理能力：

1. 授权模型产品化：把“用户本轮允许做什么”从自然语言提示升级为结构化 TaskSpec，包括可访问路径、允许收件人、允许 API host、允许工具、可接受副作用和有效时间。系统应先生成可解释的授权边界，再按边界裁决工具调用。
2. 审批缓存与最小授权：审批模式不应每次重复打断用户。用户批准一次后，只缓存同一会话、同一资源、同一工具、同一副作用范围内的最小授权，超出范围重新审批。
3. 正常业务画像：从已放行的真实正常操作中学习常见路径根、API host、邮件域、工具参数规模和调用顺序，用于降低普通读文件、健康检查、代码编辑、目录查看等正常工作的误拦。
4. 污点传播细化：把污点从“有/无风险”细化为来源、置信度、传播路径和可影响的 sink 类型。例如外部网页内容可以参与摘要，但不能直接驱动敏感文件读取、邮件外发、配置写入或 shell 执行。
5. LLM-Judge 分级调度：生产环境不建议所有请求都全量调用 LLM-Judge。更合理的做法是低风险请求只走确定性规则，中风险请求走轻量 judge，高风险 sink 或多源污点影响时走强 judge 并要求审批。
6. 正常边界样例回归集：把误拦样例沉淀为“正常边界回归集”，每次策略调整都同时测试攻击保护率和正常放行率，防止只追求拦截率导致系统不可用。
7. 可解释告警压缩：同一次工具调用可能命中多条规则，前端应聚合成“主原因 + 证据链 + 建议动作”，避免用户看到大量重复 finding 后难以判断。
8. 运行时隔离增强：能落地时继续补系统级控制，例如容器沙箱、只读挂载、网络 egress allowlist、低权限执行用户、seccomp/AppArmor/eBPF 监控；这样即使应用层误放，系统层仍可限制破坏范围。

## 反“应试化”优化记录

为降低系统只针对公开样例字段或关键词生效的风险，当前版本补充了以下通用机制：

1. 语义动作图：遍历任意工具参数树，抽取“敏感源、本地读取、外部 sink、网络外发、持久化、提权副作用、伪装正常能力、编码内容”等结构化特征，不依赖 `_hidden_payload` 等固定字段名。
2. 编码与混淆规范化：除 base64/hex 外，补充 URL percent 编码、Unicode 转义、零宽字符和分隔符规整，降低简单编码绕过概率。
3. 能力声明一致性：对“天气查询/日历/翻译/摘要”等表面正常 Skill，如果代码同时出现本地敏感读取和外部网络发送，会判定为能力声明与实际行为不一致。
4. 会话行为基线：系统只从已放行操作学习常见 API host、收件人、路径根和参数规模；后续同一会话出现目标突变或参数形态异常时触发审批提示。
5. 误报修正：普通长期经验写入不再因为“memory_write 本身是持久化”而被误拦，只有内容出现越权、敏感外发、启动项、系统配置或低信任来源覆盖未来行为时才阻断。

## 当前限制

- eBPF 系统级硬拦截在当前用户级 OpenClaw 服务中不可用。健康检查显示 `/sys/fs/bpf` 对该用户服务不可访问，且 `unprivileged_bpf_disabled=2`。当前系统仍启用了应用层系统预执行策略。
- R-Judge、AgentLAB / Agent-SafetyBench 本轮主要作为方法参考，没有直接混入工具执行前拦截指标。它们更适合后续用于多轮审计解释、风险轨迹评分和长期任务环境评测。
- 部分危险 Payload 真实进入监督链路，但 destructive shell 命令没有直接执行，这是为了避免实验破坏主机；报告中的结果来自执行前真实策略裁决和审计记录。
