# 玄鉴 功能、算法流程与答辩讲解稿

更新时间：2026-07-03

答辩口径：历史 840 条是公开样例的本地映射回归，只能说明已知集回归表现；独立盲测、正常业务集和原生 AgentDojo 必须分栏展示，未运行的数据明确写 `not_run`。

## 一句话介绍

玄鉴是面向工具型智能体的运行时安全治理系统。它不只判断一句话像不像攻击，而是判断“不可信数据是否正在驱动高权限工具动作”。

答辩时可以这样讲：

> 我们把智能体安全问题建模为“身份、信任和数据流”问题。系统在工具执行前，把用户授权任务、上下文来源、工具参数、历史行为和系统安全边界统一成策略裁决，输出 allow、ask 或 deny，并把证据实时展示到监督端。

## 系统流程

1. 用户或 OpenClaw 产生请求。
2. 系统生成 TaskSpec，记录当前任务允许的工具、目标、收件人、路径和 API host。
3. 外部内容、工具结果、记忆、Skill 和配置进入信任标签模块。
4. 污点传播记录低信任内容是否影响后续工具参数。
5. 语义动作图遍历工具参数，抽取敏感源、本地读取、外部 sink、网络外发、持久化和提权副作用。
6. ABAC、Memory Guard、工具边界域的 System Preflight、行为基线和 LLM-Judge 语义复核共同输出 finding。
7. 策略引擎给出 `allow / ask / deny`。
8. 监督端展示工具决策、告警、阻断原因、风险向量、拓扑和统计。

更完整的技术说明见 `reports/technical_design_and_algorithms.md`。

## 核心算法流程

### 1. 任务授权：先定义“能做什么”

玄鉴先把用户任务转成可检查的授权边界，而不是让模型自己理解边界。

示例：

- 用户说“总结网页”，授权的是读取网页和生成摘要。
- 用户说“发项目进度给 user@example.com”，授权的是指定收件人邮件。
- 用户说“保存报告到 notes/report.md”，授权的是普通工作区写入。

后续如果出现读取 `.env`、访问 `~/.ssh`、发到外部邮箱、写启动项、修改 OpenClaw 配置，就属于越权或意图漂移。

### 2. 信任标签：给上下文做来源护照

每段进入上下文的数据都会被打标签：

- 来源：用户、网页、邮件、PDF、图片、工具返回、记忆、配置、Skill、Webhook。
- 完整性：trusted、user、workspace、external、tainted。
- 机密性：public、internal、secret。
- 风险向量：提示注入、隐藏内容、敏感数据、外泄、持久化、工具劫持、权限、意图漂移、供应链。

这一步解决的是“模型看到的文本不是同一种东西”。用户明确指令、网页内容、第三方 Skill、历史记忆不能享有同样权限。

### 3. 污点传播：不是污染就全拦，而是看流向

污点定义来自两部分：

- 来源低信任：外部网页、邮件、PDF、图片、Webhook、第三方工具返回。
- 内容危险：隐藏指令、敏感路径、外发意图、持久化意图、网关覆盖、Skill 本地读加外联。

系统为污点生成画像，记录允许用途和禁止 sink。

例子：

- 外部网页可以被总结。
- 但外部网页不能驱动 `read_file ~/.ssh/id_rsa`。
- 外部网页也不能驱动 `send_email` 或 `call_api` 把配置发出去。

所以系统判断的是“不可信数据是否正在流向高风险工具”，而不是看到不可信数据就把整个会话废掉。

### 4. 语义动作图：从任意参数里抽行为组合

语义动作图会递归遍历工具参数，先做 URL 编码、Unicode 转义、base64、hex、零宽字符等规范化，再抽取动作节点。

关键节点：

| 节点 | 例子 |
|---|---|
| 敏感源 | `.env`、OpenClaw 配置、SSH 私钥、token |
| 本地读取 | `readFileSync`、`cat`、`open()` |
| 外部 sink | 外部 URL、webhook、非白名单邮箱 |
| 网络外发 | `fetch`、`curl`、`requests.post`、upload |
| 持久化 | `MEMORY.md`、Skill、cron、systemd、startup |
| 提权副作用 | sudo、chmod、iptables、rm -rf |
| 伪装能力 | “天气查询”却读取本地配置并外联 |

判断逻辑不是“有没有 attacker 这个词”，而是看行为组合：

- 敏感源 + 本地读取 + 外部 sink = 数据外泄。
- 正常能力声明 + 本地读取 + 网络外发 = 恶意 Skill。
- 持久化 + 审批绕过/敏感源/外部 sink = 记忆或配置投毒。

### 5. Memory Guard：记忆写入、存储、读取都检查

记忆防护分三步：

1. 写入前检查：普通偏好可写；审批绕过、默认外发、保护键修改、系统角色伪造会被阻断。
2. 存储时生成护照：来源类别、信任等级、上下文哈希、内容 SHA-256、风险向量、保护键、HMAC 签名。
3. 读取时校验：如果内容被篡改、低信任记忆试图变成未来权威规则、或成为离群高风险记忆，就要求审批或阻断。

这个机制对应记忆投毒防御里的来源标记、分层信任、关键决策绕过记忆、异常路径检测。

### 6. LLM-Judge：只做二道复核，不做唯一裁决

LLM-Judge 参与四个环节：

- 消息和工具结果：看是否有注入、越狱、凭据外泄或恶意指令。
- 工具调用：看动作是否偏离任务，是否包含越权、外泄、破坏。
- 记忆写入：看是否适合作为长期记忆。
- 溯源扫描：看 Skill、配置、工作区基础文件是否有恶意能力。

Judge 输出低/中/高风险和理由。系统把它转成 finding，再和确定性策略、污点流、ABAC、Memory Guard、工具边界域的 System Preflight 合并。这样即使模型误判，也不会绕过硬规则；即使模型不可用，系统仍然有基础防护。

当前三种调度：

- `full`：全部调用 Judge，适合离线评测。
- `risk-tiered`：低风险走确定性策略，高风险才调用 Judge，适合生产化。
- `off`：完全不调用 Judge，适合无模型基线。

## 可以重点讲的功能

### 1. 工具调用前硬裁决

每个文件读写、shell、邮件、API、记忆写入、Skill 安装都会先进入策略链。危险动作不会等执行后再审计，而是在执行前阻断。

示例：

- 用户只要求总结网页，后续出现 `read_file ~/.ssh/id_rsa`：deny。
- 用户要求发送邮件给 `user@example.com`：allow。
- 用户要求写普通报告到 `notes/report.md`：allow。
- 请求写 `startup/config.txt` 或 `openclaw.json`：deny。

### 2. 信任标签和污点传播

系统为内容标注来源和可信度：

- 外部网页、邮件、PDF、图片默认低信任。
- 含隐藏指令、敏感路径、外发暗示时升级为 tainted。
- 如果 tainted 内容影响邮件、API、shell、记忆或配置写入，就会触发 ABAC 阻断。

这能解释为什么“读网页”可以通过，但“网页诱导发密钥”会被拦。

### 3. 语义动作图

这是降低“应试化”的关键改进。系统不只匹配 benchmark 字段，而是从任意参数中抽取动作关系：

| 语义节点 | 示例 |
|---|---|
| 敏感源 | `.env`、`openclaw.json`、SSH 私钥、token |
| 本地读取 | `fs.readFileSync`、`cat`、`open()` |
| 外部 sink | 外部 URL、webhook、非白名单邮箱 |
| 网络外发 | `fetch`、`curl`、`requests.post`、upload |
| 持久化 | MEMORY、startup、cron、systemd、Skill |
| 提权副作用 | sudo、chmod、iptables、rm -rf |
| 伪装能力 | “天气查询”却读取本地配置并外联 |

判断逻辑：如果一条工具调用同时具备“敏感源 + 本地读取 + 外部 sink/网络发送”，即使没有 `attacker`、`secret` 这类直白词，也会被拦截。

### 4. Memory Guard

记忆安全不是只看关键词。系统为每条记忆写入生成 passport：

- 来源等级。
- 内容哈希。
- 风险向量。
- 保护键标记。

普通经验可以写入；会改变未来审批、路由、安全边界或默认行为的记忆会被拦截。

### 5. 行为基线

系统只从已放行操作学习本会话正常行为：

- 常用 API host。
- 常用邮件收件人。
- 常用路径根。
- 参数大小和形态。

如果同一任务突然切换到新外部 host 或参数体积异常，系统会要求审批。这个机制不是硬编码样例，而是会话内自适应。

### 6. LLM-Judge 的定位

LLM-Judge 是第二道动态复核，不是唯一防线。

最新复测采用 `risk-tiered`：benchmark 全部传入 `semanticJudge=default`，由服务端按风险分层决定是否调用 DeepSeek Judge。结果显示，风险分级比 full Judge 更接近生产形态：保持 0 漏放，同时把 full Judge 下的 7 条误拦降到 0 条。

## 对应比赛要求

| 比赛要求 | 玄鉴实现 |
|---|---|
| 提示注入、越狱、数据泄露、滥用风险 | 外部内容注入、工具响应污染、敏感源到外部 sink 流向检查 |
| 工具调用劫持 | MCP 元数据检测、网关 URL 覆盖检测、恶意 Skill 行为不一致检测 |
| 记忆中毒 | Memory Guard、passport、保护键、哈希完整性、低信任来源阻断 |
| 工具调用、代码执行、文件访问审计 | `tool_decision`、`tool_result`、`guard_finding`、`alert` |
| 允许/拒绝/询问策略 | `allow / ask / deny`，并支持 observe/approval/block 模式 |
| 实时展示告警和阻断 | `8765/security-screen`、`8765/records` |

## 答辩亮点

- 不是只做提示词过滤，而是做工具执行前的数据流治理。
- 不是只靠 LLM，主要防线是可解释的确定性策略、信任标签、污点传播、ABAC 和系统预执行。
- 不是简单全拦，普通邮件、文件、API、记忆和低风险系统观察可以放行。
- 有公开 benchmark 支撑，也有开启/关闭对比实验。
- 有 full Judge 和 risk-tiered 对比，能说明为什么生产环境不应所有请求都交给 LLM。
- UI 展示使用真实事件聚合，不造假数据。

## 当前限制怎么讲

可以直接说明：

- 当前 eBPF 因用户级服务权限不可用，页面如实显示 unavailable；应用层预执行策略仍在线。
- 危险 shell 样例不实际破坏主机，实验验证的是执行前真实裁决。
- 生产环境需要继续积累更多正常业务样本，优化白名单和审批流程。
